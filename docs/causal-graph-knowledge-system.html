叙事因果图与知识作用系统

这份交互说明面向人类阅读：它把需求文档里的抽象内容拆成几个可点击模块，解释“知识如何改变世界”，以及“如何把上层因果关系显式化、图形化、可校验化，再编译成现有系统”。

1. 为什么要做这套上层抽象

当前问题不是缺少 flag、tag、action，而是逻辑上有因果关系的内容散落在不同场景、对话、道具和条件里。内容越多，作者越难回答一个简单问题：这条传闻到底影响了哪些东西？

核心方向：编辑因果，编译变量。
过去

在场景 A 打听传闻，手动设置变量，再让场景 B 的道具条件读取这个变量。短期能跑，长期会乱。

现在要做

把“传闻 → 世界变化 → 交互变化 → 阶段推进”的因果链显式画出来。

底层不推翻

运行时仍然可以使用现有 scenario、phase、flag、condition、action，只是由上层图编译生成。

2. 知识如何作用：三态系统

知识不是背包里的道具，也不是菜单上的技能。它是一种认知能力：获得知识后，世界会变得更可读，角色会自然想起传闻，少数关键时刻才出现主动动作。

入眼

世界呈现变化。新线索显形，旧场景变深。

入心

角色或系统提示变化。帮玩家想起传闻。

入手

少量关键情境中，玩家主动照规矩办。

知识分级

C 级

只改变描述。用于世界厚度和理解变化。

高频低成本
B 级

显形线索。让隐藏痕迹、路径、异常变得可读。

调查推进
A / S 级

触发强提示或主动动作。只给主线关键知识。

少而重

4. 从上层因果图编译到现有系统

上层图是给作者看的；底层数据是给运行时吃的。不要让人类长期维护机器变量，机器变量应该由编译器生成。

底层编译结果

编译原则

  • 作者编辑语义节点和因果边，不直接维护自动变量。
  • 编译器生成 flag、condition、action、scenario phase。
  • 生成数据必须能回溯到上层节点，方便 debug。
  • 编译前做校验：孤立节点、死链、循环依赖、重复写入等。

5. 铁盒第一夜：最小验证链路

先不要做完整宏大工具。用一条最小链验证抽象是否成立。

茶馆听到香灰传闻 → 获得香灰藏事 → 城隍庙香灰层显形 → 供桌下灰痕可调查 → 铁盒第一夜阶段推进。
玩家体验

玩家不会看到“设置变量成功”。他只会感觉茶馆闲话开始影响城隍庙，旧场景显出新线索。

作者编辑

作者直接看见这条因果链，知道传闻影响哪个场景层、哪个交互点、哪个剧情阶段。

三个知识作用示例

吊死的人夜里别看脸

入眼：脚、影子、绳结成为重点。入心：关二狗想起传闻。入手：侧身看脚、看影子、正面看。

香灰不能扫尽

入眼：香灰断痕显形。入心：关二狗觉得灰不对。入手：暂不主动使用，先作为调查线索。

纸钱可能引人

入眼：纸钱方向显形。入心：意识到不是乱撒。入手：决定是否沿纸钱追查。
