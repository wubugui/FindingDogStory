层次叙事系统编辑器

这份页面用来展示编辑器应该是什么样：不是让作者手动拆 quest、scenario、flag、action，而是在多层状态图中统一编辑 Flow、实体状态覆盖、状态迁移、补丁和运行时投影。

1. 编辑器的核心目标

这个编辑器不是替代现有场景、对话、任务、Action 编辑器，而是在它们之上增加一层统一的叙事状态图视图。

作者编辑的是状态、迁移、实体补丁和叙事因果；底层系统负责执行、表现和存档。
不再手动拆流程

不用先在脑子里想完整流程，再拆成 quest、scenario、flag、condition、action。

从散装配置到总图
实体本体不动

Hotspot、NPC、Zone 仍由场景编辑器部署。新系统只在实体之上叠加叙事状态覆盖层。

不推翻 scene
必须有运行时

状态进入、退出、迁移、补丁应用不能只靠静态编译，必须由 NarrativeStateManager 管理。

不是纯编辑器

最终工作流

创建 Flow → 选择参与实体 → 给实体定义叙事状态 → 连接状态迁移 → 定义 patch / onEnter / onExit / response → 预览运行时结果 → 编译/投影到旧系统。

5. 运行时预览：状态如何真正生效

这是必须新增 NarrativeStateManager 的原因：状态变化要能立即应用 patch，并且传播到其他子图。

1
小游戏完成

Minigame 发出 complete 事件。

2
Flow 状态进入：捞箱子已完成

NarrativeStateManager 设置 Flow 子图状态。

3
状态传播到 NPC 子图

Flow.捞箱子已完成 触发 npc_ringboy 的迁移：事件前/已交互 → 事件后。

4
执行 NPC 状态 onEnter 与 patch

应用动画、巡逻、交互响应等补丁。

5
玩家之后再交互

NPC 实体逻辑读取当前叙事状态，进入对应 DialogueGraph 分支或响应配置。

运行时伪代码

sendEvent("minigame.dock_crate.complete") → setState(flow_dock, "crate_done") → transition(npc_ringboy, "before|interacted" → "after_crate") → applyPatch(npc_ringboy.after_crate.patch) → mirrorFlag("narrative.npc_ringboy.after_crate.active", true)
